# PRD: 스팸 번호 대량 차단 iOS 앱

## 1. 제품 개요

**프로젝트명**: SpamBlocker (가칭)
**플랫폼**: iOS 전용 (iPhone 11 이상, iOS 16+)
**핵심 목표**: iOS Call Directory Extension을 활용하여 **안정적으로 최대한 많은 전화번호를 차단**하는 앱
**차별화 포인트**: 경쟁앱(070 스팸패턴차단 등) 대비 **물량 압도** — 안정성을 유지하면서 차단 가능한 번호 수의 한계를 극대화

---

## 2. 배경 및 문제 정의

### 2.1 현재 상황
- 070, 080, 1588 등 스팸 전화가 일상적으로 발생
- 기존 경쟁앱들은 차단 번호 수에 한계가 있음
- iOS Call Directory Extension은 **메모리 ~15MB 제한**이라는 기술적 제약 존재

### 2.2 이전 프로젝트(new_block) 교훈
- 전략5: mmap + 10개 Extension으로 1,000만 번호 차단 시도
- 핵심 인사이트: Extension당 메모리 한계, 다중 Extension 전략의 가능성과 한계 파악
- **이번 프로젝트는 이전 코드를 참고만 하고 새로 설계**

---

## 3. 타겟 사용자

- 스팸 전화를 자주 받는 일반 사용자
- 특정 번호대(070, 080, 1588 등)의 대량 차단을 원하는 사용자
- 기술적 이해가 낮아도 간편하게 설정할 수 있어야 함

---

## 4. 핵심 기능 요구사항

### 4.1 번호대 기반 차단 설정 (MVP)
| 항목 | 설명 |
|------|------|
| 차단 단위 | **번호대(prefix) 단위** 선택 |
| 차단 방식 | Call Directory Extension — **Blocking 전용** (Identification 미사용) |
| 사용자 설정 | 약 1,000개 번호대 중 사용자가 원하는 번호대를 토글하여 선택 |
| 예시 | 070-0000 ~ 070-9999 (각각 1만 번호), 080-xxxx, 1588-xxxx 등 |

### 4.2 번호대 구성 예시
```
070-0000-0000 ~ 070-0000-9999  → 번호대 "070-0000" (10,000개)
070-0001-0000 ~ 070-0001-9999  → 번호대 "070-0001" (10,000개)
...
070-9999-0000 ~ 070-9999-9999  → 번호대 "070-9999" (10,000개)
→ 070 전체 = 10,000 번호대 × 10,000개 = 1억 개
```

사용자는 이 중에서 원하는 번호대를 선택하여 차단 범위를 커스터마이즈

### 4.3 Extension 구조 (실험적)
| 항목 | 설명 |
|------|------|
| Extension 수 | **실험을 통해 최적 수량 결정** |
| 목표 | 안정적으로 동작하는 최대 차단 번호 수 달성 |
| 메모리 전략 | 다양한 방식(mmap, 스트리밍, 직접 로드 등)을 **실험적으로 테스트** |

### 4.4 화이트리스트 (향후)
- v1에서는 미포함
- 향후 업데이트로 특정 번호를 차단 예외 처리하는 기능 추가 예정

## 5. 기술 스펙

### 5.1 개발 환경
| 항목 | 선택 |
|------|------|
| 언어 | Swift |
| UI 프레임워크 | **SwiftUI** (성능 우선, 네이티브 최적화) |
| 최소 iOS | **iOS 16** (iPhone 11 지원) |
| 아키텍처 | 새로 설계 (이전 프로젝트 참고) |

### 5.2 핵심 기술 과제
1. **Extension 메모리 한계 극복**: ~15MB 제한 내에서 최대 번호 수 등록
2. **다중 Extension 최적화**: Extension 개수별 안정성/성능 실험
3. **번호 정렬**: Call Directory API는 오름차순 정렬 필수 → 효율적 정렬/저장 전략
4. **App Group 공유**: 메인앱 ↔ Extension 간 설정/데이터 공유

### 5.3 메모리 전략 실험 계획
아래 방식들을 **단계적으로 실험**하여 최적 조합을 찾음:

| 전략 | 설명 | 예상 장단점 |
|------|------|-------------|
| **A. 직접 배열 로드** | Int64 배열을 메모리에 올려서 순회 | 간단하지만 메모리 한계 빨리 도달 |
| **B. mmap 파일 매핑** | 디스크의 바이너리 파일을 메모리 매핑 | 메모리 효율적, 이전 프로젝트에서 검증 |
| **C. 스트리밍 방식** | 파일에서 번호를 순차적으로 읽으며 등록 | 메모리 최소 사용, I/O 속도 이슈 가능 |
| **D. 하이브리드** | 위 방식들의 조합 | 실험 결과에 따라 결정 |

### 5.4 Extension 수 실험 계획
| 실험 | Extension 수 | 번호 수/Extension | 총 번호 수 |
|------|-------------|-------------------|-----------|
| 실험 1 | 1개 | 최대치 탐색 | ? |
| 실험 2 | 3개 | 최대치 × 3 | ? |
| 실험 3 | 5개 | 최대치 × 5 | ? |
| 실험 4 | 10개 | 최대치 × 10 | ? |
| 실험 5 | 10개+ | 한계 탐색 | ? |

→ 각 실험에서 **크래시 없이 안정적으로 등록 완료되는 최대치**를 기록

## 6. 데이터 전략

### 6.1 v1: 로컬 생성
- 사용자가 번호대를 선택하면 앱이 해당 범위의 전화번호를 **로컬에서 자동 생성**
- 서버 의존 없이 오프라인 동작
- 번호대 선택 → Int64 변환 → 정렬된 바이너리 파일 저장 → Extension이 읽음

### 6.2 향후: 서버 연동 (미정)
- 실제 스팸 신고 DB 기반 차단 리스트 다운로드
- 주기적 업데이트
- 구체적 계획은 v1 이후 결정
