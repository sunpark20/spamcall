# Architecture Decision Records

## 철학
MVP 속도 최우선. 070 차단 하나만 확실하게 동작하면 된다. 불필요한 기능, 추상화, 설정을 만들지 않는다.

### ADR-001: CallKit Call Directory Extension 사용
**상태**: 확정

**맥락**: iOS에서 수신 전화를 차단하는 방법이 필요하다. 1억개 번호를 시스템 레벨에서 차단해야 한다.

**검토한 대안**:
| 방법 | 설명 | 기각 이유 |
|------|------|-----------|
| CallKit CXCallDirectoryExtension | iOS 공식 전화 차단 API | **채택** |
| VPN 기반 네트워크 필터링 | NEFilterDataProvider로 VoIP 패킷 차단 | 셀룰러 통화에 적용 불가. VoIP 앱 통화만 가능. 배터리 과다 소모 |
| MDM 프로파일 | 기업용 기기 관리로 번호 차단 | 개인 사용자에게 MDM 설치 요구는 비현실적. App Store 배포 불가 |
| 통신사 API 연동 | KT, SKT 등 통신사의 스팸 차단 서비스 활용 | 통신사별 API 다름, 일반 개발자 접근 불가, 통신사 종속 |

**결정**: CXCallDirectoryExtension을 사용한다.

**이유**:
- iOS에서 셀룰러 수신 전화를 차단할 수 있는 유일한 공식 API
- 시스템 레벨에서 동작하여 배터리/성능 영향 없음 (번호 매칭은 iOS가 처리)
- App Store 심사 통과 가능 (정식 API 사용)
- 사용자 프라이버시 침해 없음 (통화 내용 접근 불가, 번호 매칭만 수행)

**트레이드오프**:
- 패턴/와일드카드 차단 미지원 → 1억개 번호 개별 등록 필요
- 익스텐션당 등록 한도 존재 → 다중 익스텐션 아키텍처 필요 (ADR-002)
- 사용자가 Settings에서 수동으로 익스텐션을 켜야 함 → UX 비용

**리스크**:
- Apple이 향후 API를 변경하거나 다중 익스텐션 패턴을 금지할 가능성 → 낮음 (WideProtect 선례)
- iOS 업데이트로 한도가 변경될 가능성 → 있음. K 값 조정으로 대응

### ADR-002: 다중 익스텐션 + 공유 소스 아키텍처
**상태**: 확정

**맥락**: 익스텐션당 등록 한도가 100만~200만개 사이로, 1억개를 단일 익스텐션에 등록할 수 없다. 번호를 분할해서 여러 익스텐션에 나눠 등록해야 한다.

**검토한 대안**:
| 방법 | 설명 | 기각 이유 |
|------|------|-----------|
| 단일 익스텐션 + 증분 로딩 | isIncremental로 여러 세션에 걸쳐 등록 | 증분 로딩도 동일한 per-extension 한도 적용. 총량 제한은 동일 |
| N개 익스텐션, 각각 별도 소스 | 익스텐션별로 하드코딩된 범위 | N개 소스 파일 유지보수 불가. K 변경 시 전체 수정 필요 |
| N개 익스텐션, 공유 소스 + 번들 ID 기반 구간 계산 | 하나의 소스 파일, 런타임에 자기 구간 결정 | **채택** |
| N개 익스텐션, 공유 소스 + App Group UserDefaults | 앱이 각 익스텐션에 구간 정보를 전달 | 불필요한 복잡성. 번들 ID만으로 충분 |
| N개 익스텐션, 공유 소스 + 빌드 설정 매크로 | SWIFT_ACTIVE_COMPILATION_CONDITIONS로 구간 주입 | xcodegen에서 타겟별 매크로 관리 복잡. 번들 ID 파싱이 더 단순 |

**결정**: N개 Call Directory Extension 타겟이 단일 `CallDirectoryHandler.swift`를 공유하고, 각 익스텐션이 런타임에 `Bundle.main.bundleIdentifier` 접미사로 자기 슬라이스 인덱스를 결정한다.

**이유**:
- 유지보수 포인트 1개 (소스 파일 하나)
- K 값 변경 시 상수 하나만 수정
- 익스텐션 수 변경 시 project.yml과 generate 스크립트만 수정
- 외부 의존성(App Group, UserDefaults 등) 없음
- 런타임 파싱 로직이 단순하고 테스트 가능

**트레이드오프**:
- 사용자가 Settings에서 N개 익스텐션을 각각 수동으로 켜야 함. N이 50이면 50번, 100이면 100번 토글
- 앱 바이너리에 N개 익스텐션이 포함되어 앱 크기 증가 (각 익스텐션은 수 KB 수준이므로 총 증가량은 미미)
- 번들 ID 파싱 실패 시 차단이 동작하지 않음 → 파싱 실패를 명시적 에러로 처리하여 방어 (0으로 폴백하지 않음)

**선례**: WideProtect (App Store 등록, 최대 80개 익스텐션 사용)

**리스크**:
- Apple이 앱당 익스텐션 수에 상한을 설정할 가능성 → 현재까지 없음. WideProtect가 80개로 운영 중
- App Store 심사에서 다수 익스텐션을 문제삼을 가능성 → 낮음. 심사 노트에 기술적 필요성 설명

### ADR-003: xcodegen으로 프로젝트 생성
**상태**: 확정

**맥락**: N개(50~100개) 익스텐션 타겟을 Xcode 프로젝트에 포함해야 한다. 수동 관리는 비현실적이다.

**검토한 대안**:
| 방법 | 설명 | 기각 이유 |
|------|------|-----------|
| Xcode GUI로 수동 생성 | File > New > Target 반복 | 50~100회 수동 작업. 실수 가능성 높음. K 변경 시 전체 재작업 |
| project.pbxproj 직접 편집 스크립트 | Python/Ruby로 pbxproj 파싱 후 타겟 추가 | pbxproj 포맷이 복잡하고 문서화되지 않음. 깨지기 쉬움 |
| xcodegen (project.yml) | YAML로 프로젝트 구조 선언, xcodegen이 xcodeproj 생성 | **채택** |
| Tuist | Swift DSL로 프로젝트 정의 | xcodegen보다 무거움. 이 프로젝트에는 과도 |
| Swift Package Manager (SPM) only | Package.swift로 전체 관리 | App Extension 타겟을 SPM만으로 정의하기 어려움 |

**결정**: xcodegen의 `project.yml`로 프로젝트를 선언적으로 정의한다. `generate_extensions.sh`가 N개 익스텐션의 디렉토리/Info.plist를 생성하고, `project.yml`이 이를 참조한다.

**이유**:
- YAML로 N개 타겟을 선언적 정의 가능
- `.xcodeproj`를 gitignore하고 `project.yml`만 버전 관리 → diff가 깔끔
- K 값 변경 → N 변경 → `generate_extensions.sh` 재실행 + `xcodegen generate`로 즉시 반영
- 설치: `brew install xcodegen` 한 줄

**트레이드오프**:
- xcodegen 의존성 추가. 빌드 환경에 설치 필요
- `.xcodeproj`가 생성물이 되어 git에 포함되지 않음 → 클론 후 반드시 `xcodegen generate` 실행 필요
- xcodegen의 YAML 문법 학습 필요 (단, 우리 프로젝트 구조는 단순)

**에러 케이스**:
- xcodegen 미설치 시 빌드 불가 → README에 설치 방법 명시
- project.yml 문법 오류 시 → xcodegen이 에러 메시지 출력. YAML 검증 후 수정
- generate_extensions.sh 미실행 시 Info.plist 누락 → xcodegen generate 실패. 스크립트 먼저 실행 안내

### ADR-004: 단계적 한도 테스트 전략 (per-extension + system-wide)
**상태**: 확정

**맥락**: 두 가지 별개의 한도를 확인해야 한다:
1. **익스텐션당 한도 K**: 단일 익스텐션이 한 번에 등록 가능한 최대 건수 (100만~200만 사이)
2. **시스템 전체 한도 T**: 모든 익스텐션의 차단 번호 합산 총량에 대한 iOS 시스템 한도 (미지)

기존 앱이 ~2000만개(20개 익스텐션 × 100만)로 운영하는 것은 검증됐다. 1억개는 5배 규모로, 아무도 검증한 적 없는 미답의 영역이다.

**핵심 인사이트**: reload 성공 ≠ 차단 동작. 시스템이 번호를 받아들여도 내부 DB/인덱스에 반영하지 못하면 전화가 차단되지 않는다. 모든 테스트의 최종 판정은 **실제 070 통화 차단 여부**로만 한다.

**결정**: Phase 0을 두 단계로 나눈다.

**Phase 0-A: 익스텐션당 한도 K 이진 탐색**
```
익스텐션 1개로 테스트.
시작: low = 1,000,000 / high = 2,000,000
반복:
  mid = (low + high) / 2
  테스트: mid개 번호 등록 시도
    성공 (reload error==nil AND 실제 통화 차단 확인) → low = mid
    실패 → high = mid
  |high - low| < 50,000 이면 종료
결과: K = low × 0.9 (안전 마진 10%)
```

**Phase 0-B: 시스템 전체 한도 T 단계적 탐색**
```
K를 확정한 후, 익스텐션 수를 단계적으로 증가:

Step 1: 20개 × K = ~2000만 (기존 앱 수준. 반드시 성공해야 함)
  → 실제 통화 차단 확인
  → 저장 공간 기록
  
Step 2: 40개 × K = ~4000만
  → 실제 통화 차단 확인 (시작/중간/끝 구간)
  → 저장 공간 기록
  → 전화 앱 성능 체감

Step 3: 60개 × K = ~6000만
  → 동일 체크

Step 4: 80개 × K = ~8000만
  → 동일 체크

Step 5: 100개 × K = ~1억 (목표)
  → 동일 체크
  → 기기 재시작 후 차단 유지 확인
  → 24시간 후 차단 유지 확인
```

**각 Step의 성공 기준** (모두 충족해야 다음 Step으로):
1. 모든 익스텐션 reload에서 error == nil
2. 차단 구간의 시작/중간/끝 번호로 발신 시 차단됨
3. 비차단 번호(010-xxxx-xxxx)는 정상 수신
4. 전화 앱의 체감 성능 저하 없음 (수신 화면 표시 지연 < 1초)
5. 기기 재시작 후에도 차단 유지

**특정 Step에서 실패한 경우**:
- 이전 Step의 총량이 시스템 전체 한도 T
- T 이내에서만 운영. 070 전체 대역이 아닌 주요 대역만 선별 차단으로 전환
- 또는 iOS 업데이트를 기다려 재시도

**이유**:
- 1억개를 한 번에 시도하면, 실패 시 어디서 깨졌는지 알 수 없음
- 단계적 증가로 정확한 한계점을 특정 가능
- 각 단계마다 실제 통화 테스트를 포함하여 "등록은 됐으나 차단 안 됨" 케이스를 잡아냄

**트레이드오프**:
- Phase 0에 상당한 시간 투자 (각 Step마다 전체 reload + 통화 테스트)
- 하지만 본 프로젝트 구축 후 실패하는 것보다 훨씬 효율적

**에러 케이스**:
- 테스트 기기에서 100만개도 실패하면? → low를 50만으로 낮춰 재시작
- Step 1(2000만)에서 실패하면? → 기기/iOS 문제. 기기 초기화 또는 다른 기기에서 재시도
- 특정 Step에서 reload는 성공하나 차단 안 되면? → 그 Step의 총량이 시스템 실질 한도
- Phase 0-B는 테스트 프로젝트(LimitTest)를 확장하여 수행. 본 프로젝트를 구축할 필요 없음

### ADR-005: E.164 번호 형식 및 범위 결정
**상태**: 확정

**맥락**: 070-XXXX-XXXX를 CallKit에 등록하려면 E.164 Int64 형식으로 변환해야 한다.

**결정**: 한국 국가코드 +82, 070의 0을 제거하여 +8270으로 변환. 전체 범위는 +827000000000 ~ +827099999999.

**변환 규칙**:
```
국내 형식:  070 - XXXX - XXXX  (11자리)
국제 형식:  +82 - 70 - XXXX - XXXX
E.164 Int64: 8270XXXXXXXX

예시:
  070-1234-5678 → +827012345678 → 827012345678 (Int64)
  070-0000-0000 → +827000000000 → 827000000000 (Int64)
  070-9999-9999 → +827099999999 → 827099999999 (Int64)
```

**범위 검증**:
- 시작: 827,000,000,000
- 끝: 827,099,999,999
- 개수: 827,099,999,999 - 827,000,000,000 + 1 = 100,000,000 (1억) ✓
- Int64 범위 내: 최대값 9,223,372,036,854,775,807 >> 827,099,999,999 ✓

**에러 케이스**:
- 070의 0을 포함하여 +820700000000으로 변환하면 → 잘못된 번호. 실제 070 수신 전화와 매칭되지 않음
- 국가코드 누락 (70XXXXXXXX) → CallKit이 국가코드 포함 E.164만 인식
- 이 변환은 코드에 상수로 하드코딩되므로 런타임 에러 가능성 없음. 단, 상수값이 정확한지 코드 리뷰에서 반드시 검증

### ADR-006: 병렬 reload 전략
**상태**: 확정

**맥락**: N개 익스텐션을 reload할 때 순차 vs 병렬 전략을 결정해야 한다.

**검토한 대안**:
| 방법 | 설명 | 기각 이유 |
|------|------|-----------|
| 완전 순차 (1개씩) | for-loop으로 하나씩 reload → 완료 대기 → 다음 | N=50이면 50 × 30초 = 25분. 너무 느림 |
| 완전 병렬 (N개 동시) | 모든 익스텐션 동시 reload | 시스템 리소스 과부하 → 타임아웃/메모리 에러 증가 가능 |
| 제한적 병렬 (M개씩) | TaskGroup + maxConcurrentOperationCount | **채택** |

**결정**: Swift Concurrency의 TaskGroup으로 병렬 reload하되, 동시 실행 수를 M(기본값 5)으로 제한한다.

**이유**:
- 완전 순차는 사용자 대기 시간이 너무 길다
- 완전 병렬은 시스템 리소스를 과도하게 사용하여 개별 익스텐션의 실패율이 높아질 수 있다
- M=5는 경험적 안전값. Phase 0 테스트에서 최적값 조정 가능

**에러 핸들링**:
- 개별 실패는 전체를 중단하지 않음 (fail-fast 아님)
- 실패한 익스텐션 목록을 수집하여 UI에 표시
- 사용자가 실패 항목만 선택적으로 재시도 가능
- 전체 재시도도 가능 (이미 성공한 익스텐션은 재등록해도 무해 — 덮어쓰기)

**엣지 케이스**:
- reload 중 앱이 백그라운드로 이동: 시스템이 이미 시작된 익스텐션 reload는 계속 처리. 아직 시작하지 않은 reload는 포그라운드 복귀 후 계속
- reload 중 앱이 종료: 진행 중이던 개별 reload는 시스템이 처리. TaskGroup은 소실. 앱 재실행 후 상태 확인 → 필요시 재시도

### ADR-007: 1억개 실패 시 폴백 전략
**상태**: 확정

**맥락**: Phase 0-B에서 시스템 전체 한도 T가 1억 미만으로 확인될 수 있다. 그때 프로젝트를 어떻게 진행할지 미리 결정한다.

**결정**: 시스템 한도 T에 따라 3단계 폴백을 적용한다.

**폴백 단계**:

| 시나리오 | 전략 | 상세 |
|----------|------|------|
| T ≥ 1억 | 원래 목표 달성 | 070 전체 대역 1억개 차단 |
| 5000만 ≤ T < 1억 | 고빈도 스팸 대역 우선 | KISA 스팸 신고 통계, 통신사 데이터 등으로 실제 스팸에 사용되는 070 하위 대역을 식별. T개 이내에서 최대 커버리지 |
| 2000만 ≤ T < 5000만 | 기존 앱 수준 + 차별화 | 주요 대역만 차단. 기존 앱 대비 차별점 모색 |
| T < 2000만 | 프로젝트 목표 재정의 | 070 전체 차단이 불가능하다는 것이 확인됨. 기존 앱과 동일 수준으로 전환하거나 프로젝트 중단 |

**"고빈도 스팸 대역 우선" 구현 시**:
- 070-XXXX-XXXX 중 실제 할당된 국번 목록은 한국인터넷진흥원(KISA) 또는 과기정통부 자료로 확인 가능
- 미할당 국번은 차단 목록에서 제외하여 총량 절약
- 이 경우 차단 목록이 고정값이 아닌 데이터 기반이 되어 아키텍처 변경 필요 (App Group을 통한 데이터 전달 등)
- MVP 범위를 넘으므로, T < 1억이 확인된 후에만 검토

**이유**:
- 사전에 폴백을 결정해두면 Phase 0 결과에 즉시 대응 가능
- "1억이 안 되면 실패"가 아니라 "한도에 맞춰 최선의 결과"를 추구

### ADR-008: 최소 iOS 버전 16.0
**상태**: 확정

**맥락**: CXCallDirectoryExtension은 iOS 10부터 사용 가능하지만, 최소 지원 버전을 결정해야 한다.

**결정**: iOS 16.0을 최소 버전으로 설정한다.

**이유**:
- iOS 16부터 SwiftUI의 안정성이 크게 향상 (NavigationStack, .onChange 등)
- iOS 16 미만 기기는 2024년 기준 점유율 5% 미만
- Swift Concurrency(async/await, TaskGroup)의 안정적 지원은 iOS 15부터이나, iOS 16이 더 안정적
- 최소 버전을 낮추면 호환성 테스트 부담 증가. MVP에서는 불필요

**트레이드오프**:
- iOS 15 이하 사용자는 이 앱을 사용할 수 없음 → 대상 사용자 중 극소수로 추정

### ADR-009: 차단 동작 검증 방법
**상태**: 확정

**맥락**: reload 성공과 실제 차단 동작은 별개다. 특히 1억개 규모에서는 시스템이 조용히 실패(silent failure)할 수 있으므로, 실제 통화로 검증하는 방법이 필수적이다.

**검토한 대안**:
| 방법 | 설명 | 평가 |
|------|------|------|
| reload API 결과만 확인 | error == nil이면 성공으로 간주 | 불충분. silent failure를 감지 못 함 |
| 다른 070 전화로 직접 발신 | 테스트 대상 기기에 070 번호로 전화를 걸어 차단되는지 확인 | **채택 (1순위)** |
| 테스트 번호 대역으로 대체 | 070 대신 발신 가능한 번호 대역(예: 010)을 임시로 차단 등록하여 테스트 | **채택 (보조)** |
| 시스템 로그 확인 | Console.app으로 CallKit 관련 시스템 로그 확인 | 보조적으로 유용. 차단 동작의 직접 증거는 아님 |

**결정**: 두 가지 방법을 병행한다.

**방법 1: 010 테스트 전화로 차단 확인 (1순위)**
- 보유 중인 010-6572-8791 전화기에서 테스트 iPhone으로 발신
- E.164: `821065728791` — 이 값은 070 범위(`827000000000`)보다 작으므로 오름차순 규칙에 맞게 070 번호들 앞에 등록 가능
- 구현: CallBlock000 익스텐션의 `beginRequest`에서 070 범위 등록 전에 `addBlockingEntry(withNextSequentialPhoneNumber: 821065728791)` 1회 호출
- 차단되면: 발신 측에서 "통화 연결 불가" 또는 바로 끊김. 테스트 iPhone의 최근 통화에 부재중 없음
- 차단 안 되면: 테스트 iPhone에서 수신 화면 표시
- **Phase 0 전 단계에서 이 방법으로 차단 동작 확인**

**방법 2: 070 VoIP 발신 (보충, 보유 시)**
- 070 번호 발신이 가능한 인터넷전화 서비스가 있다면 추가 검증
- 070 대역 시작/중간/끝 번호로 발신하여 전체 범위 커버리지 확인
- 현재 미보유이므로 방법 1이 주력

**방법 3: 시스템 로그 확인 (디버깅용)**
- Xcode > Console 또는 macOS Console.app으로 iPhone 연결 후 "CallDirectory" 관련 로그 필터링
- 등록된 번호 수, 인덱싱 성공/실패 여부를 확인할 수 있는 경우가 있음
- Phase 0에서 활용하여 시스템 내부 동작 이해

**010 테스트 번호 등록 시 주의사항**:
- `821065728791`은 070 범위 밖이므로 본 프로젝트의 차단 대상이 아님
- LimitTest(Phase 0)에서만 사용. 본 프로젝트(SpamCall070)에서는 등록하지 않음
- 테스트 완료 후 이 번호를 차단 목록에서 제거해야 함 (익스텐션 재reload로 해결)

**에러 케이스**:
- 010 번호 차단은 되지만 070 번호가 실제로 차단되는지 확신 불가 → 010 차단이 동작하면 iOS 시스템의 차단 메커니즘 자체는 정상. 070 번호도 동일 메커니즘이므로 동작해야 함. 단, E.164 변환 오류(ADR-005의 rangeBase 상수)는 별도 검증 필요
- 차단은 되지만 간헐적으로 안 되는 경우 → 시스템 부하, 인덱스 불완전 등. 반복 테스트로 재현율 측정
- 발신 측에서 차단 여부를 명확히 구분하기 어려운 경우 → 테스트 iPhone의 최근 통화 목록 확인. 차단된 전화는 최근 통화에 나타나지 않음
