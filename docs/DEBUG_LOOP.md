# 디버깅 자동화 루프 설계

## 목적
사용자로부터 에러 리포트가 접수되면, Claude가 자동으로 분석 → 수정 → 빌드 검증하는 하네스 워크플로우.

## 워크플로우

```
┌─────────────────────────────────────┐
│  1. 에러 리포트 접수                  │
│     사용자가 리포트 텍스트 붙여넣기      │
│     GoNoGo: 파싱 가능 형식인가?        │
└──────────────┬──────────────────────┘
               ▼
┌─────────────────────────────────────┐
│  2. 리포트 파싱                      │
│     기기 정보, 에러 코드, 실패 패턴     │
│     GoNoGo: 필수 필드 모두 추출?       │
└──────────────┬──────────────────────┘
               ▼
┌─────────────────────────────────────┐
│  3. KNOWN_ERRORS.md 매칭             │
│     알려진 패턴? → 사용자 조치 안내     │
│     미지 패턴? → Step 4로            │
│     GoNoGo: 전부 분류 완료?           │
└──────────────┬──────────────────────┘
               ▼
┌─────────────────────────────────────┐
│  4. 코드 조사 + 수정                  │
│     에러 원인 분석                    │
│     수정 코드 작성                    │
│     빌드 검증 (xcodebuild)           │
│     GoNoGo: 빌드 성공 + CRITICAL 위반 없음? │
└──────────────┬──────────────────────┘
               ▼
┌─────────────────────────────────────┐
│  5. 사용자에게 보고                   │
│     수정 내용 요약                    │
│     실기기 테스트 요청                 │
│     GoNoGo: 수정 요약 + 테스트 방법 명확? │
└──────────────┬──────────────────────┘
               ▼
┌─────────────────────────────────────┐
│  6. 사용자 확인                       │
│     통과 → 커밋                      │
│     실패 → Step 4로 루프             │
│     GoNoGo: "차단 확인됨" 또는        │
│     "여전히 실패" 명확 답변?           │
└─────────────────────────────────────┘
```

## 리포트 파싱 규칙

ERROR_REPORT.md에 정의된 형식을 파싱한다. 각 섹션별 추출 규칙:

### `[기기]` 섹션

```
[기기]
모델: iPhone 15 Pro
iOS: 26.3.1
저장공간: 42GB / 128GB
```

| 필드 | 추출 규칙 | 필수 |
|------|-----------|------|
| 모델 | `모델:` 뒤 텍스트 trim | Y |
| iOS | `iOS:` 뒤 텍스트 trim | Y |
| 저장공간 | `저장공간:` 뒤 `남은GB / 전체GB` 파싱 | N |

### `[익스텐션]` 섹션

```
[익스텐션]
활성화: 55 / 58
미활성: Block 042, Block 043, Block 044
```

| 필드 | 추출 규칙 | 필수 |
|------|-----------|------|
| 활성화 수 | `활성화:` 뒤 `X / 58`에서 X (정수) | Y |
| 미활성 목록 | `미활성:` 뒤 쉼표 구분 리스트. 각 항목은 `Block XXX` 형식 | N (0이면 없음) |

### `[Reload 결과]` 섹션

```
[Reload 결과]
성공: 52 / 58
실패:
  Block 007 — loadingInterrupted: 로딩 중단됨
  Block 021 — sqlite:19: SQLITE_CONSTRAINT
  Block 033 — unknown: 알 수 없는 오류
소요: 18분 32초
```

| 필드 | 추출 규칙 | 필수 |
|------|-----------|------|
| 성공 수 | `성공:` 뒤 `X / 58`에서 X (정수) | Y |
| 실패 목록 | `실패:` 다음 줄부터 들여쓰기된 라인들. 각 라인: `Block XXX — 에러코드: 메시지` | Y |
| 소요 시간 | `소요:` 뒤 텍스트 | N |

실패 라인 파싱 정규식:
```
^\s+Block\s+(\d{3})\s+—\s+(.+?):\s+(.+)$
```
- 그룹 1: 익스텐션 번호 (3자리)
- 그룹 2: 에러 코드 (예: `loadingInterrupted`, `sqlite:19`, `unknown`)
- 그룹 3: 에러 메시지 (한국어 설명)

### `[앱 상태]` 섹션

```
[앱 상태]
로딩 완료: false
버전: 1.0 (1)
```

| 필드 | 추출 규칙 | 필수 |
|------|-----------|------|
| 로딩 완료 | `로딩 완료:` 뒤 `true`/`false` | Y |
| 버전 | `버전:` 뒤 텍스트 | N |

## 각 단계별 GoNoGo 기준

### Step 1: 접수

| 조건 | 통과 | 불통과 |
|------|------|--------|
| `=== SpamCall070 에러 리포트 ===` 헤더 존재 | Go | 사용자에게 리포트 재생성 요청 |
| `[기기]`, `[Reload 결과]` 섹션 존재 | Go | 사용자에게 리포트 재생성 요청 |
| 텍스트가 빈 문자열 | NoGo | "에러 리포트를 붙여넣어 주세요" 안내 |

### Step 2: 파싱

| 조건 | 통과 | 불통과 |
|------|------|--------|
| 기기 모델 + iOS 버전 추출됨 | Go | 사용자에게 기기 정보 직접 확인 요청 |
| 에러 목록 1건 이상 추출됨 | Go | 성공 58/58이면 "에러 없음" 보고 후 종료 |
| 앱 상태(로딩 완료, 버전) 추출됨 | Go | 경고 출력 후 계속 진행 (비필수) |

### Step 3: 매칭

| 조건 | 통과 | 불통과 |
|------|------|--------|
| 모든 에러가 "알려진 에러" 또는 "미지 에러"로 분류됨 | Go | 파싱 실패한 라인 있으면 사용자에게 원본 확인 요청 |

### Step 4: 수정

| 조건 | 통과 | 불통과 |
|------|------|--------|
| `xcodebuild build` 성공 | Go | 빌드 에러 수정 후 재시도 |
| CRITICAL 규칙 위반 없음 (아래 참조) | Go | 수정 롤백 후 다른 접근 |

CRITICAL 규칙 체크리스트:
- [ ] 모든 익스텐션이 `CallBlockBase/CallDirectoryHandler.swift` 하나를 공유하는가? (복사본 생성 금지)
- [ ] 차단 번호가 E.164 Int64 오름차순인가?
- [ ] addBlockingEntry 루프에서 힙 할당(Array, String 등) 없는가?

### Step 5: 보고

| 조건 | 통과 | 불통과 |
|------|------|--------|
| 수정 내용 요약이 "에러 분석 결과" 형식으로 작성됨 | Go | 형식 맞춰서 재작성 |
| 코드 수정이 있으면 실기기 테스트 방법이 명시됨 | Go | 테스트 방법 추가 |
| 사용자 조치만으로 해결되면 조치 목록이 명확 | Go | 조치 목록 보강 |

### Step 6: 확인

| 조건 | 통과 | 불통과 |
|------|------|--------|
| 사용자가 "차단 확인됨" 등 성공 답변 | 커밋 후 완료 | - |
| 사용자가 "여전히 실패" 등 실패 답변 | Step 4로 루프 | - |
| 사용자가 애매한 답변 | NoGo | "차단이 정상 동작하나요?" 재확인 요청 |

## 에러 분류 알고리즘

각 실패 항목에 대해 순서대로 실행:

```
for each 실패_항목 in 실패_목록:
    1. 에러 코드를 KNOWN_ERRORS.md 테이블과 매칭
       - CallKit 에러: 에러코드 문자열 → 코드 번호로 변환 → "CallKit 에러" 테이블 조회
         예) "loadingInterrupted" → 코드 2
       - SQLite 에러: "sqlite:숫자" 형식 → "SQLite 에러" 테이블 조회
         예) "sqlite:19" → 코드 19 (CONSTRAINT)
       - 시스템 이슈: 증상 기반으로 "시스템 동작 이슈" 테이블 대조

    2. 매칭 성공:
       - "코드 수정" 컬럼 확인
         - "불필요" → 분류: "알려진_사용자조치"
                     → 사용자 조치 안내만 출력
         - "필요" 또는 구체적 수정 내용 → 분류: "알려진_코드수정"
                     → Step 4(코드 수정)으로 전달

    3. 매칭 실패:
       - 분류: "미지_에러"
       - 미지 에러 조사 프로토콜(아래) 실행
       - Step 4로 전달
```

## 미지 에러 조사 프로토콜

KNOWN_ERRORS.md에 없는 에러가 발견되면 아래 순서를 따른다:

```
1. 에러 도메인 확인
   - "com.apple.CallKit" → CallKit 프레임워크 에러
   - "com.apple.callkit.database.sqlite" → SQLite/DB 에러
   - 기타 도메인 → 시스템 에러

2. 에러 코드로 공식 문서 검색
   - Apple Developer Documentation: CXErrorCodeCallDirectoryManagerError
   - CallKit 릴리스 노트 (해당 iOS 버전)

3. 해당 코드 경로 추적
   - CallDirectoryHandler.swift: beginRequest → addBlockingEntry 흐름
   - ExtensionManager.swift: reloadExtension → 에러 핸들링 흐름
   - 에러가 발생하는 정확한 지점 특정

4. 재현 조건 파악
   - 특정 iOS 버전에서만? (리포트의 iOS 필드 참조)
   - 특정 기기에서만? (리포트의 모델 필드 참조)
   - 저장공간 부족과 관련? (리포트의 저장공간 필드 참조)
   - 특정 익스텐션 번호에서만? (번호 구간 계산 확인)

5. 수정 가설 수립 → 구현 → 빌드
   - 가설을 명시적으로 기록
   - 최소한의 수정으로 구현
   - xcodebuild 빌드 성공 확인

6. KNOWN_ERRORS.md에 새 패턴 추가
   - 필수 필드: 에러 코드, 원인, 사용자 조치, 코드 수정 필요 여부, 발견 일자, 발견 기기/iOS
   - describeError() 한국어 메시지 동시 업데이트
```

## 루프 탈출 조건

### 최대 반복 횟수

- Step 4 → Step 6 루프를 **최대 3회** 반복한다.
- 3회 수정 후에도 실패 시:
  ```
  권장 조치:
  1. 앱 삭제
  2. 기기 재부팅
  3. 앱 재설치
  4. Settings > 전화 > 차단 및 발신자 확인 에서 전체 익스텐션 활성화
  5. 앱에서 다시 로딩
  ```
  이후에도 실패하면 별도 이슈로 에스컬레이션한다.

### 코드 수정 불필요 케이스

- 모든 에러가 "알려진_사용자조치"로 분류된 경우:
  - 사용자 조치 안내만 출력하고 바로 완료
  - Step 4(코드 수정)을 건너뛴다

### 다중 기기 동일 에러

- 2대 이상의 기기에서 동일 에러가 보고되면:
  - 기기 고유 문제가 아닌 구조적 문제로 판단
  - K값 조정 또는 아키텍처 수정을 검토한다
  - CLAUDE.md의 "익스텐션당 등록 건수는 테스트로 확정된 한도 K의 90%로 설정" 규칙 재검토

## ERROR_REPORT.md 형식과 /debug-error 스킬 간 연결

```
사용자 기기에서 에러 발생
        │
        ▼
앱 내 "에러 리포트 복사" 버튼 탭
        │
        ▼
ERROR_REPORT.md 형식의 텍스트가 클립보드에 복사됨
        │
        ▼
사용자가 Claude에게 /debug-error + 리포트 텍스트 입력
        │
        ▼
/debug-error 스킬이 이 문서(DEBUG_LOOP.md)의 워크플로우를 실행
        │
        ▼
파싱 → 매칭 → 수정 → 보고 → 확인
```

- 리포트 형식의 정의: `docs/ERROR_REPORT.md`
- 리포트 생성 코드: `SpamCall070/ExtensionManager.swift`의 `generateReport()`
- 파싱 및 분석 실행: `.claude/commands/debug-error.md` 스킬
- 에러 패턴 DB: `docs/KNOWN_ERRORS.md`
- 에러 메시지 매핑: `SpamCall070/ExtensionManager.swift`의 `describeError()`

## KNOWN_ERRORS.md 업데이트 프로세스

새로운 에러가 발견되면 아래 절차를 따른다:

### 1. 추가할 필드

| 필드 | 설명 | 예시 |
|------|------|------|
| 에러 코드 | 도메인별 코드 번호 또는 이름 | `9` 또는 `sqlite:26` |
| 이름 | 에러 코드의 공식 이름 (있으면) | `unexpectedError` |
| 원인 | 발생 원인 분석 결과 | `iOS 26.4에서 새로 추가된 검증` |
| 사용자 조치 | 사용자가 수행할 조치 | `기기 재시작 후 재시도` |
| 코드 수정 | `불필요` / 구체적 수정 내용 | `불필요` 또는 `rangeBase 계산 수정` |
| 발견 일자 | 최초 발견 날짜 | `2026-04-16` |
| 발견 기기/iOS | 최초 발견 환경 | `iPhone 15 Pro / iOS 26.3.1` |

### 2. KNOWN_ERRORS.md 테이블에 행 추가

해당 에러 도메인의 테이블에 새 행을 삽입한다:
- CallKit 에러 → "CallKit 에러" 테이블
- SQLite 에러 → "SQLite 에러" 테이블
- 증상 기반 → "시스템 동작 이슈" 테이블
- 새 도메인이면 → 새 섹션 생성

### 3. describeError() 한국어 메시지 업데이트

`SpamCall070/ExtensionManager.swift`의 `describeError()` 함수에 새 에러 코드에 대한 한국어 메시지를 추가한다. 기존 패턴과 동일한 형식을 따른다.

### 4. 커밋

```
docs: KNOWN_ERRORS.md에 에러 코드 X 추가

- 원인: (원인 요약)
- 발견: (기기/iOS 정보)
```

KNOWN_ERRORS.md와 describeError() 변경은 동일 커밋에 포함한다.

## 구현: `/debug-error` 스킬

### 파일
`.claude/commands/debug-error.md`

### 트리거
사용자가 `/debug-error` + 리포트 텍스트 입력

### 스킬 동작

```markdown
1. 리포트 파싱 (위 "리포트 파싱 규칙" 섹션 참조)
   - [기기] 섹션에서 모델, iOS 버전, 저장공간 추출
   - [익스텐션] 섹션에서 활성화 수, 미활성 목록 추출
   - [Reload 결과] 섹션에서 성공/실패 수, 실패 상세 추출
   - [앱 상태] 섹션에서 로딩 완료, 버전 추출
   - GoNoGo: 필수 필드 누락 시 사용자에게 재생성 요청

2. docs/KNOWN_ERRORS.md 참조 (위 "에러 분류 알고리즘" 참조)
   - 각 에러 코드를 KNOWN_ERRORS.md와 매칭
   - 알려진 에러 + 코드 수정 불필요: 사용자 조치 안내 출력
   - 알려진 에러 + 코드 수정 필요: 수정 계획 수립
   - 미지 에러: 미지 에러 조사 프로토콜 실행

3. 코드 수정 필요 시
   - 관련 소스 파일 읽기 (ExtensionManager.swift, CallDirectoryHandler.swift 등)
   - CRITICAL 규칙 체크리스트 확인
   - 수정 코드 작성
   - xcodebuild 빌드 검증
   - 수정 내용 + 테스트 방법 보고

4. 미지 에러 (KNOWN_ERRORS.md에 없는 패턴)
   - 미지 에러 조사 프로토콜 실행
   - KNOWN_ERRORS.md에 새 패턴 추가 (위 "업데이트 프로세스" 참조)
   - describeError() 한국어 메시지 동시 업데이트
```
