# PRD: SpamCall070

## 목표
한국 070(VoIP) 대역 전화번호 1억개(070-0000-0000 ~ 070-9999-9999)를 iOS에서 원천 차단한다.

## 배경
한국의 070 번호는 인터넷전화(VoIP) 대역이다. 개통 비용이 낮아 스팸/보이스피싱에 대량 악용된다. 대부분의 일반 사용자는 070 번호로 정상 통화를 받을 일이 없으므로, 전체 대역을 차단해도 실질적 피해가 없다.

iOS는 기본 전화 앱에서 개별 번호 차단만 제공하며, 대역 차단 기능이 없다. CallKit의 Call Directory Extension API를 통해 앱이 시스템 수준에서 번호를 차단할 수 있지만, 패턴/와일드카드를 지원하지 않아 1억개 번호를 개별 등록해야 한다.

## 경쟁 현황 및 1억개 도전의 의미

### 기존 앱 현황
| 앱 | 익스텐션 수 | 추정 차단 수 | 비고 |
|-----|-----------|------------|------|
| WideProtect | ~80개 | ~2,000만 | App Store 운영 중. 알려진 최대 규모 |
| 기타 스팸 차단 앱 | ~20개 | ~2,000만 | 익스텐션당 ~100만개 |

### 1억개 = 미답의 영역
- 검증된 최대 규모: ~2,000만개 (20개 익스텐션 × ~100만)
- 우리 목표: 1억개 = **검증된 규모의 5배**
- 이것은 "더 많이 등록"하는 양적 문제가 아니라, iOS 시스템의 한계를 탐색하는 질적 도전이다

### 두 가지 별개의 한계 — Phase 0에서 모두 확인됨
1. **익스텐션당 한도 (per-extension limit)**: K = 1,750,000 확정. 시스템 시간 한도 ~30초, 처리 속도 ~83,000건/초 기준
2. **시스템 전체 한도 (system-wide limit)**: T >= 1억 확인. 58개 × 175만 = ~1억 150만건 등록 + 실제 차단 동작 확인

### 1억개에서 발생 가능한 시스템 레벨 문제 — Phase 0에서 검증 완료
- **시스템 DB 용량**: 1억 × 8바이트 = ~800MB raw data → **iOS가 정상 처리함을 확인**
- **수신 전화 매칭 성능**: 1억건 등록 상태에서 수신 전화 매칭 지연 없음 → **정상 동작 확인**
- **디바이스 저장 공간 소모**: 시스템이 내부적으로 얼마나 공간을 사용하는지 비공개. 800MB~수 GB로 추정
- **reload 성공 = 차단 동작**: Phase 0에서 reload 후 실제 통화 차단이 동작함을 확인 → **리스크 해소**

## 확정 파라미터 (Phase 0 실측)

| 파라미터 | 값 | 비고 |
|----------|-----|------|
| K (익스텐션당 등록 건수) | 1,750,000 | Phase 0-A에서 이진 탐색으로 확정 |
| N (익스텐션 수) | 58 | ceil(100,000,000 / 1,750,000) |
| maxConcurrent | 1 | 순차 reload 확정. 병렬은 SQLite lock 경합 |
| 총 등록 건수 | ~101,500,000 | 58 × 1,750,000. 070 대역 1억 + 여유분 150만 |
| Reload 소요 시간 | ~16분 (957초) | 58개 순차, 단일 익스텐션 ~17초 |
| 처리 속도 | ~83,000건/초 | addBlockingEntry 루프 기준 |
| 테스트 번호 | 010-6572-8791 | E.164: 821065728791, CallBlock000에 포함 |

## 사용자
- 070 번호로 오는 스팸/보이스피싱 전화에 시달리는 iOS 사용자
- 070 번호로 정상 통화를 받을 일이 없는 사람
- 비대상: 070 번호를 업무용(인터넷전화 회사 내선 등)으로 사용하는 사람

## 시스템 요구사항
- iOS 16.0 이상 (CXCallDirectoryExtension 안정성 기준)
- 실제 기기 필수 (시뮬레이터에서 CallKit Extension 미지원)
- 셀룰러 통화 지원 기기 (iPhone만 해당, iPad/iPod touch 제외)
- Apple Developer Account (익스텐션 코드 서명 필수)

## 핵심 기능

### F-001: 070 전체 대역 차단
- 070-0000-0000 ~ 070-9999-9999, 총 100,000,000개 번호를 Call Directory Extension으로 등록
- E.164 형식: +827000000000 ~ +827099999999 (Int64: 827000000000 ~ 827099999999)
- 익스텐션당 등록 한도(K) 제약으로 N개 익스텐션에 분할 등록
- 각 익스텐션은 자기 담당 구간만 처리 (번들 ID 기반 슬라이스)

### F-002: 익스텐션 상태 확인
- 58개 익스텐션 각각의 활성화 상태를 `CXCallDirectoryManager.getEnabledStatusForExtension(withIdentifier:)`으로 조회
- 표시 정보: 활성화된 익스텐션 수 / 전체 익스텐션 수 (예: "48 / 58 활성화")
- 상태 값: enabled, disabled, unknown

### F-003: Settings 이동 안내
- 사용자가 Settings에서 익스텐션을 켜야 함
- iOS 버전별 경로가 다름:
  - iOS 18 이하: "설정 > 전화 > 전화 차단 및 발신자 확인"
  - iOS 26: "설정 > 앱 > 전화 > 전화 차단 및 발신자 확인"
- `openSettingsURLString`이 아닌 `App-prefs:` URL scheme 후보 목록을 사용하여 설정 화면 이동 시도
- 주의: `App-prefs:` URL은 비공개 API이므로 App Store 심사에서 거부될 수 있음. 폴백으로 텍스트 안내 제공
- 주의: 로딩 중에는 시스템 부하로 "전화 차단 및 발신자 확인" 메뉴가 일시적으로 사라질 수 있음. 로딩 완료 후 자동 복구됨

### F-004: 익스텐션 순차 Reload
- "로딩 시작하기" 버튼 한 번으로 58개 익스텐션 순차 reload
- **maxConcurrent = 1 확정**: 병렬 실행 금지
  - 병렬 5개 시도: 36분 31초 + 3개 실패 (SQLite lock 경합)
  - 병렬 2개 시도: 16분 32초 (시간 단축 효과 없음)
  - 순차 1개: 16분 (957초) — 가장 안정적이고 가장 빠름
- 진행 상태 표시: 현재 reload 중인 익스텐션 수 / 전체 (예: "12 / 58 완료")
- 개별 익스텐션 reload 실패 시 해당 건만 실패 표시하고 나머지는 계속 진행
- reload 완료 후 상태 자동 갱신
- 주의: reload 중에는 앱 창을 유지해야 함 (백그라운드 전환 시 중단 가능)

### F-005: 로딩 완료 상태 저장
- reload 완료 시 UserDefaults에 `reloadCompleted = true`, `reloadDuration` 저장
- 앱 재시작 시 저장된 상태를 확인하여 Step 3(완료 화면) 바로 표시
- 중복 로딩 방지: 이미 완료된 상태에서는 reload 버튼 비활성화
- 강제 재로딩이 필요한 경우(iOS 업데이트 등)를 위한 리셋 수단 제공

### F-006: 에러 리포트 생성 (구현 예정)
- reload 실패 시 기기 정보 + 에러 상세를 구조화된 텍스트로 생성
- 포함 정보: 기기 모델, iOS 버전, 실패한 익스텐션 번호, 에러 코드, 소요 시간
- 클립보드 복사 버튼으로 간편 공유

## 사용자 플로우

### 최초 설치 후 설정 (Step 1 → 2 → 3)
```
Step 1: 설정에서 58개 ON
  1. 앱 실행 → "0 / 58 활성화" 상태 확인
  2. 안내에 따라 Settings > [전화 > 전화 차단 및 발신자 확인] 이동
     (iOS 26: Settings > 앱 > 전화 > 전화 차단 및 발신자 확인)
  3. 58개 익스텐션 토글을 모두 ON
     - 10개씩 켜고 ~10초 대기 (시스템 부하 분산)
  4. 앱으로 복귀 → "58 / 58 활성화" 상태 확인

Step 2: 로딩 시작하기
  1. "로딩 시작하기" 버튼 탭 → 번호 등록 시작
  2. 약 20분 소요, 앱 창 유지 필요 (백그라운드 전환 금지)
  3. 진행 상태: "12 / 58 완료" 형태로 표시

Step 3: 완료
  1. 전체 완료 후 "070-XXXX-XXXX 1억건 차단됨" 표시
  2. UserDefaults에 완료 상태 저장
```

### 재실행 시
```
1. 앱 실행 → UserDefaults에 reloadCompleted가 true이면 Step 3(완료 화면) 바로 표시
2. 별도 작업 불필요. reload 버튼 비활성화 상태
3. 강제 재로딩이 필요한 경우 리셋 수단 제공
```

### iOS 업데이트 후
```
1. iOS 업데이트 후 익스텐션 상태가 초기화될 수 있음
2. 앱 실행 → 상태 확인 → 필요시 Settings에서 재활성화 + Reload
```

### 앱 삭제 후 재설치 시
```
1. 앱 삭제 → 기기 재시작 → 앱 재설치 (순서 중요)
2. 재부팅 없이 재설치하면 시스템 DB 정리가 안 되어 "확인 중..." 무한 대기 발생
3. 재설치 후 Step 1부터 다시 진행
```

## 에러 케이스 및 처리

### E-001: 익스텐션 reload 타임아웃
- **원인**: 등록 건수가 시스템 허용 시간(~30초)을 초과
- **증상**: `beginRequest` 완료 전에 시스템이 익스텐션 프로세스 종료
- **사용자 영향**: 해당 익스텐션 담당 구간 차단 실패
- **앱 표시**: "CallBlock042 reload 실패" + 재시도 버튼
- **방어**: K = 1,750,000 확정 (처리 속도 ~83,000건/초 × ~30초 한도의 90% 수준). 정상 조건에서는 발생하지 않음

### E-002: 익스텐션 메모리 초과
- **원인**: addBlockingEntry 내부에서 시스템이 메모리를 과다 사용
- **증상**: 익스텐션 프로세스가 OS에 의해 강제 종료
- **사용자 영향**: E-001과 동일
- **방어**: 루프 내에서 힙 할당을 하지 않음 (Int64 값 타입만 사용). 그래도 발생하면 K 값을 줄여 익스텐션 수 증가

### E-003: 부분 활성화 (일부 익스텐션만 ON)
- **원인**: 사용자가 Settings에서 일부 익스텐션만 켬
- **사용자 영향**: 해당 익스텐션이 담당하는 번호 구간만 차단되지 않음
- **앱 표시**: "35 / 58 활성화 — 모든 070 번호를 차단하려면 58개 모두 켜야 합니다"
- **참고**: 부분 활성화 상태에서도 활성화된 익스텐션의 구간은 정상 차단됨

### E-004: 앱 삭제 후 재설치
- **원인**: 사용자가 앱을 삭제하고 다시 설치
- **영향**: 모든 익스텐션 상태 초기화. Settings에서 재활성화 + Reload 필요
- **중요**: 반드시 "앱 삭제 → 기기 재시작 → 앱 재설치" 순서로 진행해야 함
- 재부팅 없이 재설치하면 시스템 DB에 기존 데이터가 잔존하여 E-013(SQLite CONSTRAINT), E-014(DB 정리 지연) 발생
- **앱 표시**: 최초 설치와 동일한 상태 ("0 / 58 활성화")

### E-005: iOS 업데이트 후 익스텐션 비활성화
- **원인**: iOS 메이저 업데이트 시 Call Directory Extension이 비활성화될 수 있음
- **영향**: 차단이 중지됨. 사용자가 모르면 스팸 전화를 다시 받게 됨
- **앱 표시**: 앱 실행 시 비활성화 감지 → 재활성화 안내
- **한계**: 앱이 백그라운드에서 상태를 감시할 수 없으므로, 사용자가 앱을 열어야 감지 가능

### E-006: CXCallDirectoryManager API 에러
- **가능한 에러 코드**:
  - `.unknown`: 원인 불명 → "알 수 없는 오류. 기기를 재시작 후 다시 시도해 주세요"
  - `.noExtensionFound`: 번들 ID 불일치 → 빌드 설정 오류 (개발 단계에서 해결)
  - `.loadingInterrupted`: reload 중 중단 → 재시도 버튼 제공
  - `.entriesOutOfOrder`: 번호 순서 오류 → 로직 버그 (개발 단계에서 해결)
  - `.duplicateEntries`: 중복 번호 → 로직 버그 (개발 단계에서 해결)
  - `.maximumEntriesExceeded`: 한도 초과 → K 값 줄여야 함
  - `.extensionDisabled`: 사용자가 Settings에서 끔 → Settings 이동 안내

### E-007: 번들 ID 파싱 실패
- **원인**: `Bundle.main.bundleIdentifier`가 nil이거나 예상 형식과 다름
- **영향**: sliceIndex가 0으로 폴백 → 첫 번째 구간과 동일한 번호를 등록하려 시도 → 다른 구간 미차단
- **방어**: 파싱 실패 시 `context.cancelRequest(withError:)`로 명시적 실패 처리. fatalError가 아닌 에러 리포트

### E-008: 중복 reload 충돌
- **원인 A**: 사용자가 reload 진행 중에 다시 reload 버튼을 탭
- **원인 B**: 이미 로딩 완료된 상태에서 재로딩 시도 → 증분 모드에서 기존 데이터와 충돌하여 duplicateEntries 에러 발생
- **방어**: reload 진행 중에는 버튼 비활성화. 완료 후에도 UserDefaults의 reloadCompleted 플래그로 버튼 비활성화
- **시스템 동작**: CXCallDirectoryManager는 동일 익스텐션에 대한 동시 reload를 큐잉하므로 크래시는 발생하지 않으나, 불필요한 중복 작업 방지

### E-009: 저장 공간 부족
- **원인**: 기기 저장 공간이 부족하여 시스템이 차단 목록 DB를 저장 못 함
- **증상**: reload는 성공하나 실제 차단이 동작하지 않을 수 있음
- **한계**: 앱에서 직접 감지 불가. iOS 시스템 레벨 문제

### E-011: 시스템 전체 한도 초과 (1억개 고유 리스크) — Phase 0에서 해소됨
- **원인**: iOS 시스템 DB가 전체 앱/익스텐션 합산 차단 번호 수에 상한이 있을 수 있음
- **Phase 0 결과**: T >= 1억 확인. 58개 × 175만 = ~1억 150만건 등록 성공, 실제 차단 동작 확인
- **잔여 리스크**: iOS 메이저 업데이트로 한도가 변경될 가능성은 있으나, 현재 시점에서는 해소됨
- **증상 패턴** (참고용):
  - 패턴 A: 일정 수 이상의 익스텐션 reload가 조용히 실패
  - 패턴 B: reload는 성공하나 수신 전화 매칭이 동작하지 않음
  - 패턴 C: 시스템 성능 저하
- **원래 이것이 프로젝트 최대 리스크였음**: Phase 0에서 1억건 등록 + 실제 차단을 확인하여 해소

### E-012: reload 성공했으나 실제 차단 미동작
- **원인**: `reloadExtension` completion에서 error == nil이지만, 시스템이 내부적으로 번호를 인덱스에 반영하지 못함
- **구분 필요**: reload 성공은 "익스텐션이 번호를 전달했다"는 의미이지 "시스템이 이 번호를 차단에 사용한다"는 보장이 아님
- **검증**: reload 후 반드시 실제 통화 테스트로 차단 동작 확인
- **Phase 0 결과**: 1억건 등록 후 실제 통화 차단 정상 동작 확인됨. 현재 시점에서 이 리스크는 해소

### E-013: SQLite CONSTRAINT 에러 (sqlite:19)
- **원인**: 앱 삭제 후 시스템 DB에 기존 차단 데이터가 잔존한 상태에서 재설치 후 reload
- **증상**: reload 시 "INSERT OR IGNORE INTO PhoneNumber" 관련 에러 (sqlite error code 19)
- **사용자 영향**: reload 실패. 차단 번호 등록 불가
- **조치**: 앱 삭제 → 기기 재시작 → 앱 재설치 (순서 중요)

### E-014: 시스템 DB 정리 지연
- **원인**: 앱 삭제 후 iOS가 Call Directory DB를 즉시 드롭하지 않음. 1억건 이상의 row 정리에 시간 소요
- **증상**: "확인 중..." 무한 대기, `getEnabledStatusForExtension` 응답 지연
- **사용자 영향**: 앱이 익스텐션 상태를 확인하지 못해 다음 단계로 진행 불가
- **조치**: 기기 재시작으로 시스템 DB 정리 강제 완료

### E-015: 병렬 reload SQLite lock 경합
- **원인**: 다수 익스텐션이 동시에 시스템 DB에 INSERT 시도
- **증상**: 소요 시간 증가 + 일부 익스텐션 reload 실패
- **실측 데이터**:
  - 5개 동시: 36분 31초, 3개 실패
  - 2개 동시: 16분 32초, 성공이지만 시간 단축 효과 없음
  - 1개 순차: 16분, 전부 성공
- **방어**: maxConcurrent = 1 확정. 코드에서 병렬 수를 올리지 말 것

### E-016: 중복 로딩 시 duplicateEntries
- **원인**: 이미 로딩 완료된 상태에서 증분 모드로 다시 로딩 시도
- **증상**: 증분 모드에서 기존에 등록된 번호와 새로 등록하려는 번호가 충돌
- **에러**: `.duplicateEntries` (CXErrorCodeCallDirectoryManagerError)
- **방어**: UserDefaults에 로딩 완료 상태(`reloadCompleted`) 저장, reload 버튼 비활성화

### E-010: 정상 070 통화 차단
- **원인**: 사용자가 실제로 받아야 하는 070 번호 전화가 존재
- **영향**: 해당 통화가 무조건 차단됨
- **MVP 한계**: 화이트리스트 기능 없음. 이 앱은 "070 통화를 절대 받지 않는 사람"이 대상
- **안내**: 앱 설명에 "모든 070 번호가 차단됩니다. 070 번호로 정상 통화를 받아야 하는 경우 이 앱을 사용하지 마세요" 명시

## 엣지 케이스

### EC-001: Caller ID Spoofing
- 발신자가 070이 아닌 번호로 표시되도록 스푸핑한 경우 → 차단 불가
- 070이 아닌 번호에서 발신하지만 070으로 스푸핑한 경우 → 차단됨 (의도치 않은 차단 가능성 낮음)
- 이 앱의 범위 밖. 통신사 레벨에서만 대응 가능

### EC-002: 연락처에 저장된 070 번호
- iOS 동작: Call Directory Extension의 차단은 연락처 저장 여부와 무관하게 적용됨
- 연락처에 070 번호가 저장되어 있어도 차단됨
- MVP에서 예외 처리 없음

### EC-003: FaceTime / VoIP 앱 통화
- Call Directory Extension은 셀룰러/통신사 경유 통화에만 적용됨
- FaceTime, WhatsApp, KakaoTalk 등 앱 내 VoIP 통화에는 영향 없음
- 이 앱의 범위 밖

### EC-004: 듀얼 SIM / eSIM 환경
- Call Directory Extension은 기기 레벨에서 동작하므로 모든 SIM의 수신 전화에 적용됨
- 추가 처리 불필요

### EC-005: 여러 차단 앱 동시 설치
- iOS는 여러 앱의 Call Directory Extension을 동시에 활성화할 수 있음
- 각 앱의 차단 목록은 합집합으로 적용됨
- 충돌 없음. 단, 동일 번호가 여러 앱에 등록되면 중복이지만 문제 발생하지 않음

### EC-006: 070 대역의 유효 번호 범위
- 실제 KT, SK브로드밴드 등이 할당받은 070 번호는 전체 대역의 일부
- 이 앱은 미할당 번호 포함 전체 1억개를 등록함 → 오버블로킹이지만, 미할당 번호로 전화가 올 일이 없으므로 실질적 영향 없음

### EC-007: 기기 재시작 후 상태
- iOS는 기기 재시작 후에도 Call Directory Extension의 차단 목록을 유지함
- 재시작 후 별도 reload 불필요
- 단, 드물게 시스템 캐시 손상 시 차단이 풀릴 수 있음 → 앱 실행 시 상태 확인으로 대응

### EC-008: 백그라운드에서의 동작
- 이 앱은 백그라운드 실행이 필요 없음
- 차단 목록은 시스템 DB에 저장되므로 앱이 종료되어도 차단 유지
- reload는 포그라운드에서만 트리거

### EC-009: 앱 업데이트 시
- 앱 업데이트는 익스텐션 활성화 상태를 유지함 (Settings 토글 재설정 불필요)
- K 값이 변경된 업데이트의 경우 reload 필요 → 업데이트 후 앱 실행 시 reload 안내

### EC-010: Settings에서 "전화 차단 및 발신자 확인" 일시 소실
- reload 진행 중 시스템 부하로 Settings의 해당 메뉴가 일시적으로 사라질 수 있음
- 사용자가 Settings에서 익스텐션 상태를 확인하려 하면 메뉴 자체가 보이지 않음
- 로딩 완료 후 자동 복구됨. 별도 조치 불필요
- 앱 UI에서 "로딩 중에는 설정 메뉴가 일시적으로 보이지 않을 수 있습니다" 안내 고려

### EC-011: Settings 익스텐션 순서 비정렬
- iOS Settings가 익스텐션을 알파벳 순이 아닌 자체 정렬 기준으로 표시
- zero-padded display name("Block 000", "Block 001", ...)으로도 정렬 순서가 보장되지 않음
- 동작에는 영향 없음. 사용자가 특정 익스텐션을 찾기 어려울 수 있으나, 전부 켜야 하므로 순서 무관

### EC-012: 앱 재빌드 시 UI 상태 리셋
- Xcode Run으로 앱을 재설치하면 UserDefaults는 유지됨
- 단, F-005(로딩 완료 상태 저장)가 도입되기 전 빌드에서는 저장 코드가 없어 `reloadCompleted = false`
- 시스템 DB의 차단 데이터는 앱 재빌드와 무관하게 유지됨
- 이로 인해 UI는 "미완료" 상태이나 실제로는 차단이 동작하는 불일치 발생 가능

## 검증 전략: "실제로 차단되는가"

### 원칙
**reload 성공 ≠ 차단 동작**. 모든 테스트의 최종 판정은 "실제 070 번호로 전화를 걸었을 때 차단되는가"이다.

### Phase 0: 단계적 용량 테스트 — 완료

Phase 0은 두 가지를 동시에 테스트했다:
1. **Phase 0-A**: 익스텐션당 한도 K (이진 탐색) → **K = 1,750,000 확정**
2. **Phase 0-B**: 시스템 전체 한도 T (단계적 증가) → **T >= 1억 확인**

#### Phase 0-A 결과: K = 1,750,000
- 이진 탐색으로 익스텐션당 등록 가능 건수를 탐색
- 시스템 시간 한도 ~30초, 처리 속도 ~83,000건/초
- K = 1,750,000에서 안정적으로 reload 성공
- 안전 마진 포함하여 K = 1,750,000 확정 (한도의 90% 수준)

#### Phase 0-B 결과: 시스템 전체 한도 T >= 1억
- 58개 익스텐션 × 175만 = ~1억 150만건 등록 성공
- 실제 통화 차단 테스트: 010-6572-8791 → 테스트 기기로 발신 → 차단 확인
- 기기 재시작 후에도 차단 유지 확인
- 전화 앱 성능 저하 없음

#### 동시 실행 수 테스트 결과
| maxConcurrent | 소요 시간 | 결과 | 비고 |
|---------------|----------|------|------|
| 1 (순차) | 16분 (957초) | 58개 전부 성공 | **채택** |
| 2 | 16분 32초 | 성공 | 시간 단축 효과 없음 |
| 5 | 36분 31초 | 3개 실패 | SQLite lock 경합 |

#### 원래 계획과의 차이
| 항목 | 원래 추정 | 실측 결과 |
|------|----------|----------|
| K | 100만~200만 (미확정) | 1,750,000 확정 |
| N | ~100개 | 58개 |
| 동시 실행 | 5개 병렬 | 1개 순차 |
| 전체 소요 시간 | 10분~30분 | ~16분 |
| 메모리 피크 | 50MB 이하 | 수 KB |

### 각 Step에서의 체크리스트 (Phase 0에서 모두 통과)
- [x] reloadExtension completion에서 모든 익스텐션 error == nil
- [x] 010-6572-8791에서 테스트 iPhone으로 전화 → 차단 확인 (CallBlock000에 포함)
- [x] 차단 목록에 없는 다른 번호에서 전화 → 정상 수신 확인
- [x] Settings > iPhone Storage에서 시스템 저장 공간 변화 기록
- [x] 전화 앱의 체감 성능 (수신 화면 표시 지연 여부) 기록 → 지연 없음
- [x] 기기 재시작 후 차단 유지 확인 → 재시작 후 010-6572-8791에서 재발신 → 여전히 차단됨

### 발신 테스트 방법
- 테스트 발신 전화: 010-6572-8791 (보유 중인 다른 전화기)
- E.164 변환: +821065728791 → Int64: `821065728791`
- **이 값은 070 범위(827000000000)보다 작으므로, 오름차순 규칙을 위반하지 않고 070 번호들 앞에 등록 가능**
- 테스트 시: 하나의 익스텐션에서 `addBlockingEntry(821065728791)`을 먼저 호출한 뒤, 이어서 070 범위를 등록
- 010-6572-8791에서 테스트 iPhone으로 전화 → 차단되면 동작 확인
- 비차단 확인: 다른 번호(차단 목록에 없는)에서 전화 → 정상 수신 확인

### 폴백 전략 — Phase 0에서 T >= 1억 확인, 폴백 불필요

Phase 0 결과 T >= 1억이 확인되어 폴백 전략은 발동하지 않는다. 참고용으로 유지:

| 시나리오 | 대응 |
|----------|------|
| **T ≥ 1억** | **목표 달성. 전체 070 대역 차단 ← Phase 0에서 확인됨** |
| 5000만 ≤ T < 1억 | 실제 스팸에 사용되는 070 하위 대역만 선별 차단. KISA 스팸 신고 데이터 등 참고 |
| 2000만 ≤ T < 5000만 | 기존 앱과 동등 수준. 주요 070 대역만 차단 |
| T < 2000만 | 기존 앱 수준에도 못 미침. 프로젝트 목표 재정의 필요 |

- Push Notification
- 앱 내 튜토리얼 / 온보딩
- 다국어 지원 (한국어만)
